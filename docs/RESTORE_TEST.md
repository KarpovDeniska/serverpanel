# Restore Test — инструкция

Бэкап, который ни разу не восстанавливали — это **план** восстановления, а не гарантия. Раз в квартал (минимум) прогоняй этот чеклист, чтобы убедиться что архивы на Storage Box реально рабочие.

## Когда прогонять

- **Раз в квартал** (Q1 апреля, Q2 июля, Q3 октября, Q4 января — первая неделя).
- **После любого изменения структуры 1С-БД** (обновление платформы, релиз конфигурации).
- **После любой правки `backup.ps1`** или набора `sources` в `legacy-daily`.
- **После смены SSH-ключей** Storage Box или целевого сервера.

Каждый успешный/неуспешный прогон записать в журнал внизу этого файла.

## Что тестируем

- Минимум: **UNF** (1С-база) — скачать, распаковать, зарегистрировать информационную базу, открыть в 1С, войти под любым пользователем.
- Опционально (но желательно): **tools_xray** + **1c_settings** — что файлы открываются и не битые.

**НЕ тестируем на проде.** Все действия — на тестовой Windows-машине (VM / второй сервер / ноут с 1С-платформой).

## Где тестируем

Отдельной Windows-машины с 1С нет, поэтому тестируем **на том же hetzner-windows, но в отдельной папке**, не трогая прод `D:\1С\БД\UNF`. Файловая 1С-БД спокойно живёт в произвольном месте — 1cestart позволит держать обе ibase (прод + тестовую) в списке одновременно, чётко по именам.

**Правила безопасности:**
- Никогда не распаковывать в `D:\1С\БД\*` — только в `C:\restore_test\`.
- В 1cestart давать тестовой ibase имя типа `UNF-RESTORE-TEST-YYYY-MM-DD` — путать нельзя.
- Желательно: выбирать время, когда прод-1С не используется (после рабочего дня), чтобы избежать двойной работы с лицензионным сервером / Рутокеном.
- По окончании теста — **удалить тестовую ibase из 1cestart** и снести `C:\restore_test\` (см. §7).

## Что нужно

- Доступ к hetzner-windows по RDP (уже есть).
- SSH-ключ к Storage Box в `~/.ssh/serverpanel-seed/hetzner-storage...ed25519` на маке (или такой же на сервере — проще залить). Если нет — см. §0.
- `sftp.exe` / `scp.exe` — есть в Windows Server 2022 из коробки.
- Свободное место на `C:\`: ~2× от размера UNF.zip (сейчас ~3 GB для extract + ~1.5 GB для самого zip).

## 0. Подготовка SSH-ключа к SB на сервере (один раз)

Если ключа к Storage Box ещё нет на сервере — залить с мака:

```bash
# на маке
scp ~/.ssh/serverpanel-seed/<имя-файла-SB-ключа> Administrator@<IP-сервера>:C:/Users/Administrator/.ssh/storagebox_ed25519
```

В RDP на сервере проверить что файл на месте и права корректные (`icacls C:\Users\Administrator\.ssh\storagebox_ed25519 /inheritance:r /grant:r "Administrator:F"`).

## Чеклист

### 1. Скачать последний архив UNF с Storage Box

На **hetzner-windows** (по RDP) в PowerShell:

```powershell
$sbUser = "u571198"
$sbHost = "u571198.your-storagebox.de"
$sbPort = 23
$sbKey  = "$env:USERPROFILE\.ssh\storagebox_ed25519"  # см. §0
$today  = (Get-Date).ToString("yyyy-MM-dd")           # или дата которую тестируем

# список доступных daily-дат
"ls backups/daily/" | sftp -P $sbPort -i $sbKey -o BatchMode=yes "${sbUser}@${sbHost}"

# скачать UNF за выбранную дату
New-Item -ItemType Directory -Force -Path C:\restore_test | Out-Null
scp -P $sbPort -i $sbKey "${sbUser}@${sbHost}:backups/daily/$today/UNF.zip" C:\restore_test\UNF.zip
```

**Проверка**: файл есть, размер ≥ 100 MB (пустой backup = нулевой bytestream = проблема).

### 2. Распаковать

```powershell
Expand-Archive -Path C:\restore_test\UNF.zip -DestinationPath C:\restore_test\UNF -Force
Get-ChildItem C:\restore_test\UNF | Select Name, @{N='MB';E={[math]::Round($_.Length/1MB,1)}}
```

**Проверка**: видны файлы `1Cv8.1CD` (основной), `1Cv8Log\`, `1Cv8FTxt\`. Основной `.1CD` не 0 байт, размер близок к ожидаемому (на боевом ≈2.2 GB).

### 3. Зарегистрировать информационную базу в 1С

Запустить **1cestart.exe**, кнопка «Добавить» → «Добавление в список существующей информационной базы» → указать `C:\restore_test\UNF`. **Имя задать**: `UNF-RESTORE-TEST-<YYYY-MM-DD>` где дата — из какого бэкапа тестировал. Это важно чтобы не путать с боевой базой.

**Проверка**: база появилась в списке. Рядом должна быть рабочая `UNF` (или как она у тебя называется).

### 4. Открыть в 1С

Дважды кликнуть по базе в списке → войти под любым существующим пользователем.

**Проверка**:
- База открывается без ошибок «файл не найден» / «нарушена структура».
- Начальная страница рисуется.
- Заходим в документ/справочник (например, «Номенклатура» или «Контрагенты»), видим записи.
- Открываем последний созданный документ — его реквизиты соответствуют реальности на момент бэкапа.

### 5. Spot-check нескольких таблиц

- Номенклатура: количество позиций ≈ ожидаемое.
- Контрагенты: аналогично.
- Последние 5 документов продажи/закупки за месяц бэкапа — даты и суммы корректны.

### 6. Протестировать восстановление `tools_xray`

```powershell
scp -P $sbPort -i $sbKey -r "${sbUser}@${sbHost}:backups/daily/$today/tools_xray" C:\restore_test\tools_xray
Get-ChildItem C:\restore_test\tools_xray
```

**Проверка**: есть `xray.exe` + `config_xhttp.json` + `WinSW.xml`. Можно попробовать `.\xray.exe version` — должно напечатать версию.

### 7. Почистить за собой

В 1cestart: правый клик на `UNF-RESTORE-TEST-...` → **Удалить из списка** (НЕ «Удалить саму базу»!). Убедиться что боевая `UNF` осталась.

Потом в PowerShell:

```powershell
Remove-Item C:\restore_test -Recurse -Force
```

Проверить что боевая `D:\1С\БД\UNF` на месте:

```powershell
Test-Path D:\1С\БД\UNF\1Cv8.1CD  # должно быть True
```

## Что делать если тест провалился

| Что упало | Куда смотреть первым делом |
|---|---|
| scp `no such file` | Бэкап не доехал за выбранную дату. Посмотреть в Telegram алерт за эту дату. Если алерта нет — проблема с Task Scheduler, см. `Get-ScheduledTask serverpanel-backup-2` на hetzner-windows. |
| Expand-Archive падает | Zip повреждён. Проверить на SB размер файла через sftp `ls -l`; скачать ещё раз; если и на втором диске битый — zip-процесс сломан, откатить последнюю правку `backup.ps1`. |
| 1С: «файл не найден» | `Expand-Archive` развернул в подпапку — в 1С указать путь глубже (на подкаталог с `1Cv8.1CD`). |
| 1С: «нарушена структура» | VSS shadow на момент бэкапа поймал транзакцию в полуоткрытом состоянии. Крайне редко; попробовать бэкап за предыдущий день. Если повторяется — проверить что `Stage-Source` делает `Invoke-VssShadow` корректно (должна быть запись в логе `VSS shadow created for D:`). |
| Пустая база / нет пользователей | На момент бэкапа сама UNF была пустой (новый сервер / очистили вручную). Посмотреть `C:\restore_test\UNF\1Cv8.1CD` — если ~25 MB, это пустой шаблон. |
